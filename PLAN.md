## Architecture Refactor (Do First)

The council unanimously agreed on these structural changes before new features:

### 1. Extract pure grid logic from Player

Split `src/player.ts` into:
- **Pure grid logic** (no Three.js dependency): `isWalkable`, `FACING_DELTA`, `TURN_LEFT`, `TURN_RIGHT`, movement validation, grid state mutations
- **Camera/render layer**: reads player position each frame, positions camera + torch light

Why: enables unit testing, decouples game logic from rendering, unblocks every future system (enemies, HUD, level transitions).

### 2. Introduce GameState

A single `GameState` object owns all runtime state:
- Current level reference (`DungeonLevel`)
- Player grid position, facing, stats (HP/ATK/DEF/torch fuel)
- Inventory contents
- Entity list (enemies, items on ground)
- Turn phase

Owned by a new `src/game.ts`. All systems read/write through this — no scattered globals.

### 3. DungeonLevel type

The parsed JSON map becomes a typed `DungeonLevel` object that answers queries:
- `isWalkable(col, row)` — replaces raw `map[row][col] === 0`
- `getCellType(col, row)` — returns floor/wall/door/stairs/void
- `getEntitiesAt(col, row)` — returns enemies, items, triggers at that position

`buildDungeon` in `src/dungeon.ts` should return a `THREE.Group` (not add directly to scene) so levels can be disposed on transition.

## Build Order (Phases)

### Phase 1 — Foundation Refactor
- Extract `PlayerState` + grid logic into pure TS module
- Define `DungeonLevel` type and `GameState` type
- Load dungeon from external JSON file (replace hardcoded `MAP` in main.ts)
- Add Vitest, write ~15 tests for grid logic (boundary collisions, turns, strafe, out-of-bounds)
- `buildDungeon` returns `THREE.Group` instead of adding to scene directly

### Phase 2 — Visual Polish
- Apply pixelart textures to walls, floor, ceiling (TextureLoader, nearest-filter for pixel-perfect rendering)
- Expand map — more rooms, dead ends, varied layouts
- Texture variety — stone, brick, wood, moss variants per wall

### Phase 3 — Doors & Interaction
- Door cell type in JSON format
- Key-locked doors: key items consumed from inventory on use
- Switch/plate-operated doors: lever entities and pressure plate triggers
- Door states: locked, unlocked, open (visual + collision change)
- Extend `isWalkable` to handle door states

### Phase 4 — HUD
- Basic HUD overlay (DOM or canvas — wired to GameState, not globals)
- Health bar
- Torch fuel indicator
- Minimap (reveals explored cells)
- Inventory display (equipment slots + backpack grid)
- Facing direction indicator

### Phase 5 — Multi-Level Dungeons
- Multiple levels in a single dungeon JSON file
- Stairs up/down cell types
- Level transition: camera descent animation → fade to black → load new level → fade in
- Scene teardown: dispose old level's `THREE.Group`, build new one
- Torch fuel resource: drains over time/per step, light radius shrinks, refilled at sconces or by finding oil

### Phase 6 — Entities & Enemy System
- Billboard sprite system (camera-facing 2D sprites on grid)
- Enemy data in JSON metadata layer (type, stats, position, behavior)
- Enemy state machine: idle → aggro → chase → attack
- Grid pathfinding for enemy movement (move toward player)
- Turn model: player acts (on input), then all enemies act (on tween-end)
- Enemy variety: at least 2-3 types with different stats

### Phase 7 — Combat
- Combat model: TBD (decide when Phase 6 is playable)
- Damage formula: TBD
- Death/respawn: TBD
- Player equipment affecting ATK/DEF stats
- Consumable items (health potions, torch oil)
- Combat feedback: visual flash, sprite animation, damage numbers

### Phase 8 — Later Resources & Polish
- Hunger system: drains over time, food items restore it
- Sanity system: drains near enemies/darkness, visual distortion effects
- Save/load (at minimum: save on stair transitions)
- Enemy AI variety: patrol paths, line-of-sight spotting, hiding, fleeing
- Secret walls
- Ambient sound (if desired)

## Proposed File Structure

```
src/
  main.ts              — bootstrap: create renderer, start loop
  game.ts              — GameState, turn management, game loop coordination
  dungeon/
    types.ts           — Level, Cell, CellType, Entity type definitions
    loader.ts          — fetch JSON, validate, return DungeonLevel
    builder.ts         — build Three.js geometry from DungeonLevel (returns Group)
  player/
    state.ts           — PlayerState: grid position, facing, stats (pure, no Three.js)
    controller.ts      — reads input, updates PlayerState
    renderer.ts        — positions camera + torch from PlayerState
  entities/
    enemy.ts           — enemy state machine, AI, grid position
    billboard.ts       — billboard sprite renderer (camera-facing)
    item.ts            — item definitions, pickup logic
  ui/
    hud.ts             — health, resources, minimap, inventory overlay
    inventory.ts       — inventory management (equip, use, drop)
  utils/
    grid.ts            — pure grid helpers: isWalkable, pathfinding, directions
```

Not all folders needed immediately — grow into this structure as features are added.

## JSON Dungeon Format (Sketch)

Final schema decided at implementation time, but the skeleton:

```jsonc
{
  "name": "The Sunken Vault",
  "levels": [
    {
      "id": "level-1",
      "width": 11,
      "height": 5,
      "grid": [
        // 2D array of cell types
        // "W" = wall, "." = floor, "D" = door, ">" = stairs down, "<" = stairs up, " " = void
      ],
      "entities": [
        // sparse list — only cells with content
        { "row": 2, "col": 5, "type": "enemy", "props": { "kind": "skeleton", "hp": 10 } },
        { "row": 1, "col": 8, "type": "item", "props": { "kind": "key", "id": "gold-key" } },
        { "row": 3, "col": 4, "type": "door", "props": { "requires": "gold-key" } },
        { "row": 1, "col": 2, "type": "trigger", "props": { "action": "open-door", "target": "3,6" } }
      ],
      "playerStart": { "row": 2, "col": 2, "facing": "E" }
    }
  ]
}
```

## Dungeon Format — Future-Proofing Notes

Design direction captured early to inform data model evolution. Not all of these are in scope for v1, but the format should not paint us into a corner.

### 1. Text-editable grid with rich cell vocabulary

The grid should stay human-editable in a text editor (no level editor planned soon). Single chars for common cells (`.#DSU`), but the entity system handles richer semantics — locked doors, lever-triggered doors, keys, etc. are expressed as entities referencing grid positions, not as multi-char grid tokens.

### 2. Custom dungeon objects with future 3D models

Objects like fountains, altars, bookshelves, thrones should be representable as entities on walkable cells (`O` cell type + entity metadata). Initially rendered as placeholder geometry or billboards; later each gets its own 3D model. Some will be interactable (examine, use, trigger effects). The `Entity` interface with `[key: string]: unknown` index signature supports this — type-specific props can be added per object kind without schema changes.

### 3. Per-cell visual customization (ceiling heights, textures)

Different areas of a level should support varied ceiling heights and wall/floor textures. The `CellOverride` interface already exists for this (`ceilingHeight`, `wallTexture`, `floorTexture`). Implementation deferred to Phase 2+ but the data model is ready. Could also be expressed as an overlay file alongside the grid for cleaner separation.

### 4. True multi-level 3D scenes

The long-term vision goes beyond separate levels loaded one at a time. Multiple vertical levels should coexist in a single 3D scene — the player walks across levels, sees down through voids to lower floors, and can fall to a lower level by stepping into open space. This requires:
- Multiple grid layers stacked vertically with a Y offset
- Void cells that are transparent (no floor geometry) — revealing the level below
- Fall mechanic when stepping onto a void cell
- Combined scene graph with all visible levels rendered simultaneously

This is a significant architectural evolution beyond the current Phase 5 plan (which loads levels independently). Park for post-v1 or a dedicated phase.

## Testing Strategy

- **Framework**: Vitest (zero-config with Vite)
- **Scope**: Pure logic only — grid movement, collision, inventory, turn order
- **Skip**: Camera tweens, lighting, visual output, rendering
- **Target**: ~15 tests initially, grow with features
- **No CI pipeline** — just `npm test` before committing
